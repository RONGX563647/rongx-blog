第七章：GC算法 — 三种打扫策略的深度解析

上一章知道了"找垃圾"的标准（可达性分析），这一章讲"怎么清理"。三种算法对应三种场景，理解它们就理解了所有GC收集器的底层原理。

🧹 7.1 标记-清除（Mark-Sweep）—— 最原始最简单

🧹 大扫除比喻

标记阶段：清洁工走进办公室，在每个要丢的文件上贴便利贴（标记可回收对象）
清除阶段：第二次走一遍，把贴了便利贴的文件扔进垃圾桶

优点：简单，不需要搬动文件（存活对象不动）
缺点：扔完文件后柜子里像蜂窝煤——产生大量碎片。下次塞新文件时，明明总空间够，但没有一块连续的够大——分配效率降低

指标评价
速度一般（两次扫描）
空间碎片严重（最大缺点）
是否需要移动对象否

📋 7.2 标记-复制（Mark-Copy）—— 取精华去糟粕

📋 搬家比喻

与其清理旧柜子的碎片，不如直接换个新柜子：
标记阶段：在要保留的文件上贴"重要"便利贴
复制阶段：把贴了"重要"的文件搬到新柜子，整整齐齐排列
旧柜子：直接清空（格式化！）

优点：新柜子整整齐齐，零碎片，分配效率极高
缺点：浪费一半空间（新柜子必须留空着等搬入），而且存活对象多的时候搬家很累

指标评价
速度极快（只需复制存活对象）
空间碎片零碎片（最大优点）
空间浪费50%（必须有一块空区）

适用场景：新生代（对象死亡率>90%，只需复制极少数存活对象）。这解释了为什么Eden:Survivor比例是8:1:1——只有10%空间被浪费。

📐 7.3 标记-整理（Mark-Compact）—— 碎片的终结者

📐 整理书架比喻

标记阶段：在不要的书上贴便利贴
整理阶段：把要保留的书全部推到书架一边，紧挨着排列。扔掉那边全空出来了

优点：既没有碎片（都推整齐了），又不浪费空间（不需要双倍空间）
缺点：推书很累——需要移动所有存活对象并更新引用，STW时间比前两种都长

指标评价
速度慢（移动+更新引用开销大）
空间碎片零碎片
空间浪费无浪费

适用场景：老年代（存活率高，如果用复制算法要搬整个老年代，代价太大）。

🎨 7.4 三色标记法（CMS/G1/ZGC的底层）

并发GC需要标记时用户线程还在运行（不然STW太长）。传统的"标记-清除"要求STW，因为标记过程中用户线程可能修改引用关系。三色标记法解决了这个问题。

🟢🟡🔴 检查清单比喻

检查员只有一个记事本，不能同时检查所有文件。他用三种颜色标记进度：
🟢 绿色：已经检查完，没问题（存活）
🟡 黄色：检查过本身，但它的孩子还没检查（待扫描）
🔴 红色：还没检查（初始状态）

最后，所有🔴（红色）的都是垃圾。但并发标记的时候用户线程还在修改引用，可能导致"漏标"——一个本应存活的被当成垃圾。需要用读写屏障来防止。

stateDiagram-v2
    [*] --> 红色_未扫描
    红色_未扫描 --> 黄色_已标记待扫描: 从GC Roots可达
    黄色_已标记待扫描 --> 绿色_已扫描: 检查完所有子引用
    Note right of 红色_未扫描: 最终剩余的红
色对象 = 垃圾

🧠 三色口诀

红→黄→绿，剩红是垃圾

初始全红 → 从根出发变黄 → 检查完孩子变绿 → 最后还红的就是垃圾

⚡ 7.5 记忆集与卡表（Remembered Set & Card Table）

分代GC的一个问题：Minor GC时只扫描新生代，但老年代可能引用了新生代对象。如果Minor GC时扫描整个老年代找引用，那就变成Full GC了。

📇 索引卡比喻

图书馆有一种跨馆借阅索引卡——只要A馆的书被B馆的人借了，就在索引卡上记一笔。

Minor GC查新生代时，看一眼索引卡知道哪些老年代对象引用了新生代，只扫描这几页，不用翻阅整个老年代档案。

Card Table 就是这个索引卡。老年代被分成512字节的"卡片"，只要某张卡片内的对象引用了新生代，这张卡片就标记为"脏"。

🎓 本章自检

1. 标记-清除：简单但有碎片，老年代配合整理用
2. 标记-复制：零碎片但浪费空间，新生代专用
3. 标记-整理：无碎片无浪费但慢，老年代专用
4. 三色标记：并发标记的基础，红→黄→绿
5. 记忆集/卡表：跨代引用索引，避免Minor GC扫描全老年代
