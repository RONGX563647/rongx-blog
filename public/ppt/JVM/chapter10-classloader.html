第一节：类加载器的层次结构 - 从Bootstrap到自定义

类加载器的职责

  • 唯一标识：在JVM中，类由"类加载器实例 + 类全限定名"唯一确定。即使相同字节码，由不同加载器加载也会产生不同的Class对象
  • 命名空间：每个类加载器有自己的命名空间，不同命名空间中的类相互隔离，即使全限定名相同
  • 可见性规则：子加载器可以访问父加载器加载的类，但父加载器不能访问子加载器加载的类
类加载器 加载范围 Java中获取 实现类
Bootstrap $JAVA_HOME/lib 或 -Xbootclasspath 指定的核心类库（rt.jar、java.base等） String.class.getClassLoader() 返回 null C++实现（sun.misc.Launcher$ExtClassLoader的父级）
Platform JDK 9+的模块系统，加载Java SE平台模块（javax.*、java.sql.*等） ClassLoader.getPlatformClassLoader() jdk.internal.loader.ClassLoaders$PlatformClassLoader
Application 用户classpath下的类，包括应用的jar包和class文件 ClassLoader.getSystemClassLoader() jdk.internal.loader.ClassLoaders$AppClassLoader
自定义 用户自定义加载逻辑，通常加载特定路径或特定来源的类 new MyClassLoader() 继承java.lang.ClassLoader

⚠️ JDK 8 vs JDK 9+ 的重要变化

JDK 8及之前：

• Bootstrap → Extension (ExtClassLoader) → Application (AppClassLoader)

• 扩展类加载器加载 $JAVA_HOME/lib/ext 目录


JDK 9+（模块化后）：

• Bootstrap → Platform (PlatformClassLoader) → Application (AppClassLoader)

• Extension ClassLoader 被废弃，由 Platform ClassLoader 替代

• java.ext.dirs 系统属性不再被支持

第二节：双亲委派模型 - Java类加载的核心机制

// ClassLoader.loadClass() 核心源码逻辑 protected Class<?> loadClass(String name, boolean resolve) { // 1. 检查该类是否已被当前加载器加载 Class<?> c = findLoadedClass(name); if (c == null) { try { // 2. 有父加载器，委派给父加载器 if (parent != null) { c = parent.loadClass(name, false); } else { // 3. 没有父加载器，使用Bootstrap c = findBootstrapClassOrNull(name); } } catch (ClassNotFoundException e) { // 父加载器找不到，抛出异常 } if (c == null) { // 4. 父加载器加载失败，自己尝试加载 c = findClass(name); } } if (resolve) { resolveClass(c); } return c; }

✅ 双亲委派模型的三大优势

1. 避免类的重复加载

• 当父加载器已经加载了某个类时，子加载器不会再次加载，确保类在JVM中只有一个Class对象

• 节省内存，避免同一类的多个版本共存导致的混乱


2. 保证Java核心API不被篡改

• 核心类库（如java.lang.String、java.lang.Object）由Bootstrap加载，用户自定义的ClassLoader无法加载这些类

• 即使恶意用户自定义了同名类，也会被委派给Bootstrap加载，确保使用的是官方版本


3. 类加载的信任链

• 信任级别：Bootstrap > Platform > Application > 自定义

• 核心类库具有最高信任级别，安全敏感的操作只允许核心类执行

🔑 破坏场景一：SPI（Service Provider Interface）机制

问题：JDBC的java.sql.Driver接口由Bootstrap加载，但具体驱动（如MySQL的com.mysql.cj.jdbc.Driver）由Application ClassLoader加载。按照双亲委派，Bootstrap无法加载Application的类。

解决方案：引入线程上下文类加载器（Thread Context ClassLoader, TCCL）

原理：TCCL 是从上向下加载，与双亲委派方向相反，因此被称为"逆向委派"。

⚠️ 破坏场景二：Tomcat的类加载器架构

Tomcat的类加载器层次（与传统双亲委派不同）：

• 优先加载：WebApp ClassLoader 先尝试自己加载，找不到才委托给父加载器（与双亲委派相反）

• 隔离：每个WebApp有独立的ClassLoader，不同应用的同名类相互隔离

• 共享：Shared ClassLoader 加载多个WebApp共享的类（如Spring、Hibernate）

💡 TCCL 的注意事项

1. 线程池环境：在线程池中，新线程的TCCL可能继承自创建线程，需要注意上下文切换

2. Web环境：Web容器会为每个请求设置正确的TCCL（对应应用的ClassLoader），请求结束后恢复

3. 异步编程：使用CompletableFuture等异步API时，TCCL可能丢失，需要手动传递

4. 最佳实践：在异步/跨线程场景中，使用try-finally确保TCCL被正确恢复

✅ 类加载器设计最佳实践

1. 优先使用双亲委派：除非有特殊需求，不要破坏双亲委派模型

2. 明确加载范围：自定义ClassLoader应明确自己的加载范围，避免越权加载

3. 及时释放引用：热部署场景下，确保旧ClassLoader能被GC回收

4. 使用线程上下文类加载器：在SPI场景中正确使用TCCL

5. 记录加载日志：在findClass中记录日志，便于排查问题

6. 避免静态引用：ClassLoader相关的代码避免使用静态变量，防止泄漏
