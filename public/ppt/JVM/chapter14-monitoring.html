第十四章：JVM监控与诊断工具

全面掌握JVM监控与诊断工具的使用，深入理解故障排查方法论、性能调优SOP、Arthas实战技巧以及GC日志分析方法。

🔧 JDK内置工具全景图

JDK提供了丰富的命令行工具，可用于监控、诊断和调优Java应用：

1. jps - Java进程状态查看工具

用于列出所有Java进程，类似于Linux的ps命令

参数说明示例
-l输出主类全名jps -l
-v输出JVM参数jps -v
-m输出传递给main方法的参数jps -m
-V输出通过flagfile传入的参数jps -V
// 输出示例： // 12345 com.example.Application -Xms2g -Xmx2g // 12346 sun.tools.jps.Jps -l -v // 实际使用： jps -l // 12345 com.example.webapp.Application // 12346 sun.tools.jps.Jps

2. jstat - JVM统计信息监控工具

实时监控JVM的内存、GC、类加载等统计信息

参数监控内容示例
-gc垃圾回收统计jstat -gc 12345 1000 10
-gcutilGC利用率(百分比)jstat -gcutil 12345 1000
-gccapacity堆各区域容量jstat -gccapacity 12345
-gcnew新生代GC统计jstat -gcnew 12345
-gcold老年代GC统计jstat -gcold 12345
-class类加载统计jstat -class 12345
-compilerJIT编译统计jstat -compiler 12345
// jstat -gcutil 输出示例： // S0 S1 E O M CCS YGC YGCT FGC FGCT GCT // 0.00 99.12 45.67 67.89 89.12 78.45 123 5.678 2 1.234 6.912 // 0.00 99.12 46.23 67.90 89.12 78.45 124 5.689 2 1.234 6.923 // 列说明： // S0/S1: Survivor0/1使用率 // E: Eden使用率 // O: Old(老年代)使用率 // M: Metaspace使用率 // YGC/FGC: Young/Full GC次数 // YGCT/FGCT: Young/Full GC总耗时 // GCT: GC总耗时 // 监控建议： // 1. 每1秒输出一次，持续监控： jstat -gcutil 12345 1000 // 2. 关注FGC频率，如果频繁Full GC说明有问题 // 3. 老年代使用率持续增长可能预示内存泄漏

3. jmap - 内存映像工具

生成堆转储快照(dump)，查看内存使用情况

参数说明示例
-heap打印堆详细信息jmap -heap 12345
-histo打印对象直方图jmap -histo 12345
-histo:live仅统计存活对象jmap -histo:live 12345
-dump:live,format=b,file=生成堆转储文件jmap -dump:live,format=b,file=heap.hprof 12345
// jmap -heap 输出示例(G1收集器)： // Garbage-First (G1) GC with 8 thread(s) // Heap Configuration: // MinHeapFreeRatio = 40 // MaxHeapFreeRatio = 70 // MaxHeapSize = 4294967296 (4.0GB) // NewRatio = 2 // SurvivorRatio = 8 // jmap -histo:live 输出示例： // num #instances #bytes class name // 1: 123456 12345678 [B // 2: 98765 9876543 [C // 3: 45678 4567890 java.lang.String // 4: 23456 2345678 java.util.HashMap$Node // 生成堆转储文件(线上谨慎使用，会导致STW)： jmap -dump:live,format=b,file=/tmp/heap_$(date +%Y%m%d_%H%M%S).hprof 12345

⚠️ 生产环境警告

jmap -dump会触发Full GC并暂停应用，生成dump文件可能占用大量磁盘空间。线上环境建议在业务低峰期执行，或使用Arthas的heapdump命令(支持压缩)。

4. jstack - 线程堆栈追踪工具

生成线程快照，用于诊断死锁、线程阻塞等问题

参数说明示例
-l打印锁信息jstack -l 12345
-m打印native方法栈jstack -m 12345
-F强制输出(无响应时)jstack -F 12345
// jstack 输出示例： // "main" #1 prio=5 os_prio=31 tid=0x00007fb8c0001000 nid=0x1b03 runnable // java.lang.Thread.State: RUNNABLE // at java.io.FileInputStream.readBytes(Native Method) // at java.io.FileInputStream.read(FileInputStream.java:255) // at java.io.BufferedInputStream.read1(BufferedInputStream.java:284) // 线程状态说明： // NEW: 新建，尚未启动 // RUNNABLE: 运行中或就绪 // BLOCKED: 等待获取监视器锁 // WAITING: 无限期等待 // TIMED_WAITING: 限期等待 // TERMINATED: 已终止 // 导出线程快照： jstack -l 12345 > /tmp/thread_dump_$(date +%Y%m%d_%H%M%S).txt // 快速分析线程状态分布： jstack -l 12345 | grep "java.lang.Thread.State" | sort | uniq -c // 输出示例： // 10 java.lang.Thread.State: BLOCKED // 45 java.lang.Thread.State: WAITING // 3 java.lang.Thread.State: TIMED_WAITING // 2 java.lang.Thread.State: RUNNABLE

5. jcmd - 多功能诊断命令

JDK 7+引入的多功能工具，可替代jps、jstat、jmap、jstack

命令等效工具说明
jcmd -ljps -l列出Java进程
jcmd <pid> GC.heap_infojmap -heap堆信息
jcmd <pid> GC.class_histogramjmap -histo类直方图
jcmd <pid> Thread.printjstack线程快照
jcmd <pid> VM.flagsjinfo -flagsJVM参数
jcmd <pid> GC.run手动触发GC执行GC
jcmd <pid> VM.system_properties-系统属性
// 查看所有可用命令： jcmd 12345 help // 生成堆转储： jcmd 12345 GC.heap_dump /tmp/heap.hprof // 触发GC： jcmd 12345 GC.run // 查看JIT编译信息： jcmd 12345 compiler.codecache

🔍 诊断工具使用流程

1
jps -l → 获取进程ID
列出所有Java进程，找到目标应用PID
↓
2
jstat -gcutil PID 1000 → 监控GC状态
观察内存使用率、GC频率和耗时，判断是否存在内存问题
↓
3
jstack -l PID → 分析线程状态
检查是否有死锁、线程阻塞、CPU占用高的线程
↓
4
jmap -histo:live PID → 查看对象分布
找出内存占用最大的对象类型，定位可能的内存泄漏
↓
5
Arthas → 深度诊断
使用trace/watch/profiler等命令进行深度分析

🔥 CPU占用过高

使用thread -n找出CPU占用最高的线程，通过profiler生成火焰图定位热点代码。

💧 内存泄漏

使用heapdump生成堆转储文件，通过MAT分析大对象和GC Roots引用链。

♻️ GC频繁

使用jstat监控GC频率，分析GC日志，调整堆大小或选择合适的GC收集器。

🧵 线程死锁

使用jstack -l生成线程快照，查找BLOCKED状态的线程和锁竞争关系。

🔧 GC日志分析实战

// 使用脚本分析GC日志 // 1. 统计GC总次数和总耗时 grep "GC pause" gc.log | wc -l // 输出GC总次数 // 2. 找出耗时最长的GC事件 grep "GC pause" gc.log | sort -t: -k3 -n | tail -5 // 找出最慢的5次GC // 3. 计算GC频率(每小时) grep "GC pause" gc.log | awk '{print $1}' | cut -dT -f1 | sort | uniq -c // 按天统计GC次数 // 4. 使用工具分析 // GCViewer: 可视化GC日志，生成报告 // GCEasy: 在线GC日志分析平台 // IBM Pattern Modeling: 高级GC分析 // 5. 识别问题模式： // a) 频繁Minor GC → 新生代过小，对象分配过快 // b) 频繁Full GC → 老年代空间不足，可能存在内存泄漏 // c) GC时间逐渐增长 → 堆碎片化严重或内存泄漏 // d) GC后内存回收很少 → 存活对象过多，可能存在内存泄漏

性能调优SOP步骤

Step 1: 确定性能目标

明确应用的性能指标要求：响应时间、吞吐量、并发数、内存使用等

// 示例目标： // - 平均响应时间 < 200ms // - P99响应时间 < 500ms // - TPS > 1000 // - GC停顿 < 100ms // - 堆内存使用 < 70%
Step 2: 建立性能基线

使用压测工具(如JMeter、wrk)建立当前性能基线，作为优化对比基准

// 使用wrk进行HTTP压测 wrk -t12 -c400 -d30s http://localhost:8080/api/test // 输出：Requests/sec, Latency分布等 // 压测同时监控JVM指标 jstat -gcutil 12345 1000 > /tmp/jstat.log &
Step 3: 定位性能瓶颈

使用各种工具定位瓶颈：CPU、内存、GC、I/O、锁竞争等

// CPU瓶颈定位： top -Hp 12345 // 找出CPU占用最高的线程 // 转换为16进制 printf "%x\n" 12346 // 输出: 303a // 在jstack输出中查找对应线程 jstack 12345 | grep 303a -A 20 // 内存瓶颈定位： jstat -gcutil 12345 1000 // 观察内存使用趋势 // GC瓶颈定位： jstat -gc 12345 1000 // 观察GC频率和耗时 // 锁竞争定位： jstack -l 12345 // 查找BLOCKED状态的线程
Step 4: 制定优化方案

根据瓶颈类型选择优化策略：

  • CPU密集型：优化算法、减少不必要的计算、使用更高效的容器
  • 内存密集型：优化对象生命周期、减少大对象创建、使用对象池
  • GC瓶颈：调整堆大小、选择合适的GC算法、优化内存分配
  • I/O瓶颈：使用异步I/O、批量处理、缓存优化
  • 锁竞争：减少锁粒度、使用无锁数据结构、并发集合
Step 5: 实施优化并验证

逐个实施优化措施，每次变更后重新压测，验证优化效果

// 优化前记录基线数据 // TPS: 850, P99: 650ms, GC: 5次/min // 实施优化(如调整堆大小) java -Xms4g -Xmx4g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -jar app.jar // 重新压测验证 wrk -t12 -c400 -d30s http://localhost:8080/api/test // TPS: 1050, P99: 420ms, GC: 3次/min ✓
Step 6: 持续监控与优化

✅ 最佳实践总结

1. 建立完善的监控体系：Prometheus + Grafana + 告警机制
2. GC日志必须开启：保留至少7天日志，便于问题回溯
3. 定期进行性能压测：每次代码变更后验证性能指标
4. 掌握Arthas等高级工具：提高线上问题排查效率
5. 遵循调优SOP：避免盲目调整，确保每次变更可追溯
6. 建立性能基线：作为优化效果的对比基准
7. 关注业务指标：性能调优最终服务于业务目标

⚠️ 线上排查注意事项

1. 生产环境谨慎使用jmap -dump，会导致Full GC停顿
2. Arthas attach进程时会有短暂停顿
3. 开启详细GC日志会影响性能(约3-5%)
4. 线程快照(jstack)频繁执行会影响应用性能
5. 优先使用监控数据定位问题，而非直接上生产环境抓dump
6. 保留足够的磁盘空间用于日志和dump文件
