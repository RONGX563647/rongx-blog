第十二章：JIT编译技术

深入理解JVM即时编译器(Just-In-Time Compiler)的底层原理，掌握C1/C2编译器工作机制、分层编译、优化技术以及去优化策略。

为什么需要JIT？

  • 解释执行瓶颈：字节码解释执行效率低，每条指令都需翻译
  • AOT编译缺点：静态编译无法根据运行时信息优化
  • JIT优势：结合解释+编译，热点代码编译为本地机器码
  • Profile Guided Optimization：基于运行时性能数据做针对性优化

🔑 JIT的核心思想

只编译热点代码，而非所有代码。通过运行时的性能监控(Profile)，识别出执行频率高的方法或循环，将其编译为高效的本地机器码，实现"二八定律"的最优收益。

HotSpot的编译器选择

  • Client编译器(C1)：32位JVM默认，启动快，优化少，适合桌面应用
  • Server编译器(C2)：64位JVM默认，编译慢但优化激进，适合服务端
  • Graal编译器：JDK 9+引入，Java编写的编译器，支持AOT
  • 分层编译(Tiered Compilation)：JDK 8默认启用，混合使用C1和C2

分层编译的5个级别

层级说明优化程度编译速度
Level 0解释执行无优化最快
Level 1C1编译，无profiling轻量优化快
Level 2C1编译，带有限profiling中等优化中
Level 3C1编译，带完整profiling收集运行时数据中
Level 4C2编译深度优化慢

编译策略：

  • 方法从Level 0开始，根据调用计数器决定是否编译
  • Level 3收集完整的profile数据，为C2编译提供优化依据
  • 频繁执行的方法最终会升级到Level 4(C2编译)
  • 通过InvocationCounter和BackedgeCounter统计方法调用和循环回边次数

JIT编译执行流程

源码.java → javac编译 → 字节码.class → 解释执行(L0) → C1编译(L1-L3) → C2编译(L4) → 本地机器码

热点代码检测机制

  • 方法调用计数器(Invocation Counter)：统计方法被调用次数
  • 回边计数器(Back Edge Counter)：统计循环执行次数
  • 阈值计算：-XX:CompileThreshold=10000(C1)，C2基于动态计算
  • 衰减机制：计数器定期衰减，避免早期热点影响后期优化
  • OSR编译(On-Stack Replacement)：针对长循环的特殊编译，允许从循环中间开始执行编译后的代码
// 热点代码检测触发条件 // 方法调用计数器阈值计算： int methodThreshold = CompileThreshold / (Tier*2 + 1); // Level 3: 10000 / (3*2+1) = 1428次调用后触发C1编译 // Level 4: 基于profile数据，C2编译器决定是否编译 // 回边计数器用于检测循环热点： for(int i = 0; i < 1000000; i++) { // 每次循环回边，计数器+1 // 达到阈值后触发OSR编译 }

JIT核心优化技术详解

🔧 方法内联

消除方法调用开销，将小方法展开到调用处。默认阈值35字节，频繁调用325字节。

🔍 逃逸分析

分析对象作用域，实现栈上分配、标量替换、同步消除三大优化。

🔒 锁优化

锁消除、锁粗化、偏向锁，减少同步开销，提升并发性能。

⚡ 其他优化

死代码消除、公共子表达式消除、数组边界检查消除、空指针检查消除。

  • 方法内联(Method Inlining)
    • 小方法调用直接展开，消除调用开销
    • 默认内联阈值：-XX:MaxInlineSize=35字节，-XX:FreqInlineSize=325字节
    • 虚方法内联需要结合类层次分析(CH A)和内联缓存(Inline Cache)
    • 通过-XX:+PrintInlining查看内联决策
  • 逃逸分析(Escape Analysis)
    • 分析对象的作用域是否逃逸出方法或线程
    • 状态：GlobalEscape(全局逃逸)、ArgEscape(参数逃逸)、NoEscape(不逃逸)
    • 优化结果：栈上分配、标量替换、同步消除
  • 锁优化(Lock Optimization)
    • 锁消除(Lock Elimination)：JIT发现锁对象不逃逸时，直接移除synchronized
    • 锁粗化(Lock Coarsening)：合并多个连续的加锁操作
    • 偏向锁(Biased Locking)：无竞争时，减少CAS操作
  • 其他优化
    • 死代码消除(Dead Code Elimination)
    • 公共子表达式消除(Common Subexpression Elimination)
    • 数组边界检查消除(Array Bounds Check Elimination)
    • 空指针检查消除(Null Check Elimination)

场景1：标量替换优化

对象被拆解为基本类型，完全消除对象分配：

// 优化前：创建对象 public Point createPoint() { return new Point(1, 2); // NoEscape，可优化 } // JIT优化后：标量替换 // 等价于： public Point createPoint() { int x = 1; int y = 2; // 直接返回x和y，无需创建Point对象 }

场景2：同步消除

// StringBuffer的线程安全是不必要的 public String buildString() { StringBuffer sb = new StringBuffer(); sb.append("Hello"); // synchronized方法 sb.append(" World"); // synchronized方法 return sb.toString(); } // JIT会消除这些synchronized，因为sb没有逃逸

去优化(Deoptimization)机制

去优化触发流程图

  • 为什么需要去优化？
    • JIT基于当前运行时的假设进行激进优化
    • 当假设被打破时，必须回退到解释执行或重新编译
    • 保证优化的正确性，同时保持灵活性
  • 触发去优化的场景
    • 类型假设失效：内联缓存失效，发现新的子类
    • 分支预测失败：if条件概率分布发生变化
    • 空指针假设错误：之前认为不会null，现在出现了null
    • 数组类型假设错误：数组实际类型与假设不符
  • 去优化过程
    • 保存当前执行状态(寄存器、局部变量等)
    • 卸载优化后的机器码
    • 恢复到解释器或重新触发编译
    • 记录去优化信息，影响后续编译决策
// 去优化案例：类型假设失效 interface Animal { void speak(); } class Dog implements Animal { public void speak() { System.out.println("Woof"); } } class Cat implements Animal { public void speak() { System.out.println("Meow"); } } // 前期只传入Dog，JIT假设animal是Dog类型并内联 // 后期传入Cat，触发去优化，重新编译 public void makeSpeak(Animal animal) { animal.speak(); // 可能触发去优化 }

OSR(On-Stack Replacement)编译

  • 应用场景：超长循环方法，循环还未结束就已经达到编译阈值
  • 核心问题：编译完成后，如何从循环中间切入执行？
  • 解决方案：
    • 保存当前循环状态(i的值、局部变量等)
    • 跳转到编译后的代码继续执行
    • 需要额外的状态保存和恢复逻辑
  • OSR标记：方法名会显示为 methodName @ bci (字节码索引)
  • 性能影响：OSR编译的代码通常比正常编译的代码稍慢

内联缓存(Inline Cache)

  • 问题背景：虚方法调用(polymorphism)需要动态分派，性能开销大
  • 单态内联缓存(Monomorphic)：只有一种类型，直接内联调用
  • 多态内联缓存(Polymorphic)：少数几种类型，类型检查后内联
  • 超多态内联缓存(Megamorphic)：类型太多，退化为虚方法调用
  • 优化建议：保持调用点的类型稳定，避免频繁的类型变化
// 内联缓存示例 // 单态缓存：JIT会假设list的类型是ArrayList public void processList(List list) { list.get(0); // 如果一直是ArrayList，会内联调用 } // 如果调用点交替传入ArrayList和LinkedList // 会升级为多态缓存，性能下降 // 继续增加新类型会升级为超多态缓存，完全退化为虚调用

🔧 实战：JIT优化效果验证

// 测试方法内联的效果 public class InliningTest { // 小方法，会被内联 private static int add(int a, int b) { return a + b; } public static void main(String[] args) { long start = System.nanoTime(); int sum = 0; // 预热+执行 for(int i = 0; i < 100000000; i++) { sum += add(i, 1); } long end = System.nanoTime(); System.out.println("耗时: " + (end - start) + "ns"); } } // 运行参数： // java -XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining InliningTest // 查看输出确认add方法是否被内联

JIT调试参数

参数作用示例输出
-XX:+PrintCompilation打印编译信息方法名、编译层级、编译大小
-XX:+UnlockDiagnosticVMOptions解锁诊断选项开启其他JIT诊断参数

🎯 JIT优化最佳实践

1. 保持方法短小：有利于方法内联，减少调用开销

2. 避免频繁的类型变化：保持调用点的类型稳定，减少去优化

3. 预热很重要：生产环境启动后应有预热期，让JIT充分编译

4. 使用final关键字：帮助JIT做静态分派判断，减少虚方法调用

5. 循环优化：避免在循环中创建不必要的对象，利用逃逸分析优化

6. 谨慎使用synchronized：JIT可能无法优化某些复杂场景的锁

7. 监控编译线程：编译线程数默认由CPU核数决定，高并发场景需关注

编译线程与编译队列

  • 编译线程：
    • C1编译线程数：-XX:CICompilerCount，默认根据CPU核数计算
    • C2编译线程数：独立配置
  • 编译队列：
    • 方法提交到编译队列等待编译
    • 队列满时，方法会在队列中等待或放弃编译
    • 可通过-XX:ReservedCodeCacheSize控制代码缓存大小
  • 代码缓存(Code Cache)：
    • 存储编译后的本地机器码
    • 默认大小：240MB(Server JVM)
    • 缓存满时会触发Code Cache Flush，影响性能
    • 监控命令：jcmd compiler.codecache
// 编译线程数计算 // JDK 8+ 默认： // CICompilerCount = max(2, CPU核数 / 2 + 1) // 代码缓存监控 // jcmd compiler.codecache // 输出： // CodeHeap 'non-profiled nmethods': size=120MB, used=45MB // CodeHeap 'profiled nmethods': size=120MB, used=67MB // CodeHeap 'non-nmethods': size=12MB, used=3MB // 如果used接近size，说明代码缓存不足 // 可通过 -XX:ReservedCodeCacheSize=512m 增大
