📋 2.2 类加载器子系统 — 翻译公司的"前台"

🏢 前台比喻

你走进一家翻译公司，递上一份文件。前台要做三件事：
① 找到文件（从硬盘或网络上找到.class文件）
② 验证格式（检查文件是不是合法的字节码，防止伪造文件）
③ 登记入档（把类信息存入"方法区"，方便翻译官查阅）

这就是类加载的三个阶段：加载(Loading) → 验证(Verification) → 准备(Preparation)

JVM有三个类加载器，像三层前台：

类加载器负责加载位置父加载器
Bootstrap（启动类加载器）rt.jar（String, Object等核心类）JAVA_HOME/lib无（C++实现）
Extension（扩展类加载器）扩展类库JAVA_HOME/lib/extBootstrap
Application（应用类加载器）你写的代码，classpath下的类classpathExtension

💡 为什么不让你自己加载核心类？

双亲委派模型：一个类先问爸（父加载器），爸再问爷爷，最顶层的Bootstrap先加载。如果Bootstrap加载失败了，才轮到你。这样保证了String永远只有一个版本——你写的java.lang.String永远不会生效。这是JVM的"安全检查"。

📊 2.3 运行时数据区 — 翻译官的"办公室"

翻译官上班需要：一本字典（方法区）、一张办公桌（栈）、一个文件柜（堆）。

⚡ 2.4 执行引擎 — 翻译官本人

翻译官拿到一份文件，有两种翻译方式：

方式怎么做速度类比
解释执行读一行翻译一行启动快，执行慢📖 现场口译
JIT编译先全文翻译成当地语言再讲启动慢，执行快📋 同声传译

现代的HotSpot JVM是把两者结合起来——第一次执行用解释（快速开始），发现某段话被反复要求翻译（热点），就花点时间全文翻译好缓存起来（JIT），下次直接念翻译好的。

🎯 垃圾回收器也在执行引擎里

GC（垃圾回收器）实际上是一个后台线程，它时不时扫描"公用文件柜"（堆），把没人在用的文件清理掉。这个机制在第六章到第八章会详细讲。

❓ 2.5 费曼问答：初学者最常问的三个问题

Q1: 为什么方法区也在"堆"里？

有些资料说"方法区在堆里"，有些说"方法区不在堆里"。这取决于JDK版本：JDK8以前方法区叫"永久代"，是堆的一部分。JDK8以后方法区叫"元空间"，不在堆里，用的是本地内存（不占JVM堆内存）。

Q2: 栈为什么会溢出？

📝 草稿纸比喻

翻译官只有一张A4草稿纸（栈空间有限）。如果翻译的文件太复杂，需要同时记太多行——递归调用太深——草稿纸写不下了，就StackOverflow了。

Q3: 堆和栈的区别一句话？

堆 = 公用文件柜（放对象，大家共享，GC管理）
栈 = 私人草稿纸（放方法调用，线程私有，用完即弃）

🧠 一句话记忆

堆存对象，栈存方法，方法区存类

🎓 本章自检

1. JVM四大部门：类加载器（前台）+ 运行时数据区（办公桌）+ 执行引擎（翻译官）+ 本地接口（外线）
2. 类加载器分三层：Bootstrap → Extension → Application
3. 堆和方法区是共享的，栈和PC是私有的
4. 执行引擎 = 解释执行（口译） + JIT编译（同声传译）
5. 堆存对象（文件柜），栈存方法调用（草稿纸）
