第五章：API接口设计 — 前后端沟通的共同语言

费曼学习法：API就是前后端之间的"共同语言"。就像中国人和美国人交流需要英语，Vue和Spring Boot交流需要HTTP+JSON。

本章让你成为"前后端翻译官"，写出清晰、规范、防错的接口。

本章目标：看到一个URL就能秒懂它做了什么操作。

🌐 费曼类比：API = 餐厅菜单

URL = 菜名（/api/users = 用户这道菜）
HTTP方法 = 你对服务员说的话（GET = 我要看菜单，POST = 我要加一道菜，PUT = 这道菜改配方，DELETE = 退菜）
请求参数 = 备注（微辣、不要香菜、pageSize=10）
响应JSON = 端上来的菜

好的API就像好菜单——一眼看懂、结构清晰、不会端错菜。

📐 5.1 RESTful设计原则

RESTful是一种软件架构风格，通过HTTP协议实现资源的增删改查操作。

📋 5.2 API命名规范

URL设计规范

  • 使用名词而非动词：/api/users 而不是 /api/getUsers
  • 使用复数形式：/api/users 而不是 /api/user
  • 使用小写字母和连字符：/api/user-roles 而不是 /api/UserRoles
  • 资源嵌套：/api/users/{userId}/orders 表示用户的订单
  • 查询参数：用于过滤、排序、分页，如 ?status=1&page=1&size=10
// ✅ 好的API设计
GET /api/users # 获取用户列表
GET /api/users/1 # 获取ID为1的用户
POST /api/users # 新增用户
PUT /api/users/1 # 更新ID为1的用户
DELETE /api/users/1 # 删除ID为1的用户

// ❌ 不好的API设计
GET /api/getUsers # 使用动词
GET /api/user/1 # 使用单数
POST /api/createUser # 使用动词
GET /api/user?id=1 # 用查询参数代替路径参数
POST /api/users/delete/1 # 在URL中包含操作

🔢 5.4 HTTP状态码规范

状态码 含义 使用场景
200 OK 请求成功，返回数据
201 Created 资源创建成功（POST）
204 No Content 请求成功，无返回内容（DELETE）
400 Bad Request 请求参数错误
401 Unauthorized 未认证/Token无效
403 Forbidden 无权限访问
404 Not Found 资源不存在
500 Internal Server Error 服务器内部错误

🔄 5.6 接口版本控制

随着业务发展，API可能需要升级，版本控制可以保证向后兼容。

版本控制方式对比

方式 示例 优点 缺点
URL路径 /api/v1/users 简单直观，广泛使用 URL中嵌入版本
请求头 Header: API-Version: 1 URL保持简洁 不够直观
查询参数 /api/users?version=1 灵活 容易遗漏

@RequestMapping("/api/v1/users")
public class UserControllerV1 {
  // v1版本接口
}

@RestController
@RequestMapping("/api/v2/users")
public class UserControllerV2 {
  // v2版本接口（新字段、新逻辑）
}

⚠️ 5.7 错误码设计

合理的错误码体系可以快速定位问题，提高开发效率。

错误码 含义 处理方式

💡 API设计最佳实践

1. 使用RESTful风格：资源用名词，操作用HTTP方法
2. 统一响应格式：{code, msg, data} 结构
3. 合理使用状态码：2xx成功、4xx客户端错误、5xx服务端错误
4. 提供Swagger文档：降低前后端沟通成本
5. 版本控制：使用URL路径方式 /api/v1/xxx
6. 错误码体系：区分HTTP状态码和业务错误码
7. 使用Apifox/Postman：接口调试与Mock数据生成

🔗 5.9 GraphQL vs REST API 对比

特性 REST API GraphQL
数据获取 固定结构，可能过度获取 按需获取，避免多余数据
接口数量 多个端点（/users, /users/:id...） 单一端点（/graphql）
版本控制 /api/v1/users, /api/v2/users 通过字段废弃实现平滑升级
学习曲线 低，HTTP标准 中，需要学习GraphQL语法
缓存 HTTP缓存机制成熟 需第三方库支持
适用场景 CRUD系统、简单API 复杂查询、多端适配
推荐 大部分后台管理系统 数据关系复杂、移动端优先

🔒 5.10 API签名安全（防篡改 + 防重放）

对敏感API请求进行签名校验，防止参数篡改和重放攻击。

// 前端签名生成 import CryptoJS from 'crypto-js' function signRequest(params, secret) { const timestamp = Date.now() const nonce = Math.random().toString(36).slice(2) // 1. 按key字母排序参数 const sortedKeys = Object.keys(params).sort() const signStr = sortedKeys.map(k => `${k}=${params[k]}`).join('&') // 2. 拼接：signStr + timestamp + nonce + secret const raw = `${signStr}${timestamp}${nonce}${secret}` // 3. HMAC-SHA256签名 const signature = CryptoJS.HmacSHA256(raw, secret).toString() return { timestamp, nonce, signature } } // 使用 const signed = signRequest({ username: 'admin', pageNum: 1 }, 'SECRET_KEY') axios.get('/api/users', { params: { username: 'admin', pageNum: 1 }, headers: { 'X-Timestamp': signed.timestamp, 'X-Nonce': signed.nonce, 'X-Signature': signed.signature } }) // 后端签名校验拦截器 @Component public class SignInterceptor implements HandlerInterceptor { private static final long TIMEOUT = 5 * 60 * 1000; // 5分钟失效 @Override public boolean preHandle(HttpServletRequest req, HttpServletResponse resp, Object handler) { String timestamp = req.getHeader("X-Timestamp"); String nonce = req.getHeader("X-Nonce"); String signature = req.getHeader("X-Signature"); // 1. 检查时间戳（防重放） if (Math.abs(System.currentTimeMillis() - Long.parseLong(timestamp)) > TIMEOUT) { return fail(resp, "请求已过期"); } // 2. 检查nonce是否已使用（Redis存储5分钟） if (redisTemplate.hasKey("nonce:" + nonce)) { return fail(resp, "重复请求"); } // 3. 重新计算签名并比对 String expected = computeSign(req.getParameterMap(), timestamp, nonce); if (!expected.equals(signature)) { return fail(resp, "签名验证失败"); } // 4. 标记nonce已使用 redisTemplate.opsForValue().set("nonce:" + nonce, "1", Duration.ofMinutes(5)); return true; } }

🔗 5.11 SSE（Server-Sent Events）服务端推送

SSE比WebSocket更轻量，单向（服务端→客户端），适合日志推送、进度通知等场景。

// 后端SSE接口 @RestController @RequestMapping("/api/sse") public class SseController { private static final Map<String, SseEmitter> emitters = new ConcurrentHashMap<>(); @GetMapping("/subscribe/{userId}") public SseEmitter subscribe(@PathVariable String userId) { SseEmitter emitter = new SseEmitter(0L); // 永不超时 emitters.put(userId, emitter); emitter.onCompletion(() -> emitters.remove(userId)); emitter.onTimeout(() -> emitters.remove(userId)); // 发送连接成功事件 try { emitter.send(SseEmitter.event().name("connected").data("连接成功")); } catch (IOException e) { emitters.remove(userId); } return emitter; } // 推送消息给指定用户 public static void push(String userId, String event, Object data) { SseEmitter emitter = emitters.get(userId); if (emitter != null) { try { emitter.send(SseEmitter.event().name(event).data(data)); } catch (IOException e) { emitters.remove(userId); } } } } // 前端SSE连接 const eventSource = new EventSource('/api/sse/subscribe/' + userId) eventSource.addEventListener('connected', (e) => { console.log('SSE连接:', e.data) }) eventSource.addEventListener('notification', (e) => { const data = JSON.parse(e.data) console.log('收到通知:', data) }) eventSource.addEventListener('progress', (e) => { console.log('进度:', e.data) // 更新进度条 }) eventSource.onerror = () => { console.log('SSE断开，3秒后重连...') setTimeout(() => location.reload(), 3000) } // SSE vs WebSocket // SSE：单向推送、HTTP协议、自动重连、轻量、适合日志/进度/通知 // WebSocket：双向通信、独立协议、自定义重连、适合聊天/协作

🧠 费曼自测：你能解释给同学听吗？

Q1：GET、POST、PUT、DELETE分别什么时候用？

A：GET = 看菜单（不改变数据，读操作），POST = 点菜（创建新东西），PUT = 改配方（完整更新），DELETE = 退菜（删除）。记住口诀：看→点→改→退。

Q2：为什么接口要版本号（/api/v1/users vs /api/v2/users）？

A：就像APP更新。v1.0的接口老用户还在用，v2.0的新功能不能破坏老版本。版本号就像给接口贴保质期标签，新老共存互不影响。

Q3：GraphQL比REST好在哪里？

A：REST像自助餐——后端定好什么菜你就吃什么（数据不可选），GraphQL像点菜——你想吃什么就点什么（前端决定返回哪些字段）。小项目REST够了，复杂数据关系时才考虑GraphQL。
