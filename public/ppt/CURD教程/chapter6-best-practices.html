🎯 6.1 前端防抖与节流

防抖和节流是优化高频事件处理的两种常用技术。

防抖（Debounce）

在事件触发后等待一段时间，如果期间再次触发则重新计时。适用于搜索框输入。

// Vue3 Hook - useDebounce
import { ref } from 'vue'

export function useDebounce(fn, delay = 300) {
  let timer = null
  return (...args) => {
    if (timer) clearTimeout(timer)
    timer = setTimeout(() => {
      fn(...args)
    }, delay)
  }
}

// 使用示例：搜索框
const searchQuery = ref('')
const handleSearch = useDebounce((value) => {
  // 调用搜索API
  fetchSearchResults(value)
}, 300)

// 输入时触发
<el-input v-model="searchQuery" @input="handleSearch" />

节流（Throttle）

在一段时间内只允许执行一次。适用于滚动事件、窗口resize、按钮防重复点击。

// Vue3 Hook - useThrottle
export function useThrottle(fn, delay = 300) {
  let lastTime = 0
  return (...args) => {
    const now = Date.now()
    if (now - lastTime >= delay) {
      fn(...args)
      lastTime = now
    }
  }
}

// 使用示例：防止重复提交
const handleSubmit = useThrottle(async () => {
  await submitForm()
  ElMessage.success('提交成功')
}, 1000)

💾 6.4 Spring事务管理

事务保证多个数据库操作的原子性，要么全部成功，要么全部失败。

@Transactional使用要点

  • 作用范围：只能应用于public方法
  • 异常回滚：默认只对RuntimeException回滚
  • 同类调用：同一类内方法调用事务不生效（需要注入自身或使用AopContext）
  • 传播行为：默认REQUIRED（有事务加入，无事务新建）
@Service
public class UserServiceImpl implements UserService {

  @Resource
  private UserMapper userMapper;

  @Resource
  private OrderMapper orderMapper;

  // 基本事务
  @Transactional
  public void createUserWithOrder(User user, Order order) {
    userMapper.insert(user); // 新增用户
    order.setUserId(user.getId());
    orderMapper.insert(order); // 新增订单
    // 如果任意一步失败，两步都会回滚
  }

  // 指定回滚异常
  @Transactional(rollbackFor = Exception.class)
  public void complexOperation() throws Exception {
    // 所有异常都会回滚
  }

  // 只读事务（优化性能）
  @Transactional(readOnly = true)
  public List<User> queryUsers() {
    return userMapper.selectList(null);
  }

  // 手动回滚
  @Transactional
  public void manualRollback() {
    try {
      // 业务逻辑
    } catch (Exception e) {
      TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
  }
}

📊 6.5 分页优化

大数据量分页查询的性能优化方案。

深分页问题

当页码很大时（如10000页），OFFSET会扫描大量无用数据，导致查询变慢。

// ❌ 低效的深分页（MySQL会扫描100000条数据然后丢弃）
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 100000;

// ✅ 方案1：游标分页（推荐）
SELECT * FROM users WHERE id > 100000 ORDER BY id LIMIT 10;

// ✅ 方案2：子查询优化
SELECT * FROM users
WHERE id >= (SELECT id FROM users ORDER BY id LIMIT 100000, 1)
ORDER BY id LIMIT 10;

// ✅ 方案3：延迟关联
SELECT t1.* FROM users t1
INNER JOIN (SELECT id FROM users ORDER BY id LIMIT 100000, 10) t2
ON t1.id = t2.id;

MyBatis Plus游标分页实现

// 前端传递cursor（上一页最后一条记录的ID）
const queryWithCursor = {
  cursor: lastId, // 上一页最后一条ID
  pageSize: 10

// 后端实现
public PageResult<User> queryByCursor(Long cursor, Integer pageSize) {
  LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
  if (cursor != null) {
    wrapper.gt(User::getId, cursor); // 只查询ID大于cursor的记录
  }
  wrapper.orderByAsc(User::getId).last("LIMIT " + pageSize);
  
  List<User> list = userMapper.selectList(wrapper);
  return new PageResult<>(list, null); // 游标分页无法获取总数
}

🧠 费曼自测：你能解释给同学听吗？

Q1：防抖和节流有什么区别？（用电梯比喻）

A：防抖 = 电梯等人——有人按按钮就重新计时，最后一个人进来后再关门。节流 = 地铁安检——无论多少人，安检口每隔固定时间放一个人。搜索框用防抖（等用户打完字），滚动用节流（固定频率检查）。

Q2：为什么Git提交要规范？

A：就像给快递贴标签。feat(user): 新增登录功能——一看就知道是"用户模块的花瓶"而不是"厨房的锅"。git log时能一目了然，回滚时知道去哪里找。

Q3：Vite打包为什么要手动分包？

A：假设你的快递公司每次送货都要把所有仓库货物重新装车——效率极低。分包就像预设好"vue专用箱"、"工具库专用箱"，每次只更新变化的部分，加载速度从5秒降到500毫秒。
